# Task 1: Call an LLM from Code — Implementation Plan

## LLM Provider

**Provider:** Qwen Code API (развернут на VM)

**Model:** `qwen3-coder-plus`

**Причины выбора:**
- 1000 бесплатных запросов в день
- Работает из России
- Не требует кредитную карту
- Поддерживает OpenAI-совместимый API
- Сильная модель для кодинга

## Архитектура агента

### Входные данные
- Вопрос пользователя передается как первый аргумент командной строки
- Пример: `uv run agent.py "What does REST stand for?"`

### Конфигурация
- Чтение переменных окружения из `.env.agent.secret`:
  - `LLM_API_KEY` — API ключ для аутентификации
  - `LLM_API_BASE` — базовый URL API (OpenAI-совместимый)
  - `LLM_MODEL` — имя модели

### Поток выполнения
1. Парсинг аргументов командной строки (вопрос пользователя)
2. Загрузка конфигурации из `.env.agent.secret`
3. Формирование запроса к LLM API (OpenAI-совместимый формат)
4. Отправка запроса через HTTP POST
5. Парсинг ответа от LLM
6. Формирование JSON-ответа: `{"answer": "...", "tool_calls": []}`
7. Вывод JSON в stdout
8. Логирование отладочной информации в stderr

### Обработка ошибок
- Проверка наличия аргумента командной строки
- Проверка наличия файла `.env.agent.secret`
- Обработка HTTP ошибок (таймаут, 4xx, 5xx)
- Таймаут запроса: 60 секунд
- Вывод ошибок в stderr, exit code != 0

### Структура выходных данных
```json
{
  "answer": "Текст ответа от LLM",
  "tool_calls": []
}
```

## Используемые библиотеки
- `httpx` — HTTP клиент для запросов к API (уже есть в зависимостях)
- `pydantic-settings` — загрузка настроек из .env файла (уже есть в зависимостях)
- `sys` — для вывода в stdout/stderr
- `json` — для сериализации ответа
- `argparse` или `sys.argv` — для парсинга аргументов

## Тестирование
Один регрессионный тест:
1. Запуск `agent.py` как subprocess с тестовым вопросом
2. Парсинг stdout как JSON
3. Проверка наличия ключей `answer` и `tool_calls`
4. Проверка, что `tool_calls` — пустой массив
